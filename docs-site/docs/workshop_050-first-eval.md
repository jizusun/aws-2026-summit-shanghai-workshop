## 这一步在做什么

Agent 已经部署完成。在写任何评估之前，先做一件事：**亲眼看一次它的"裸表现"**。

这一步是刻意安排的对照基线——我们需要一个"优化前"的快照，等 Phase 5 改完之后用同样的问题对比。此时 Agent 对你一无所知：不知道你在哪个部门、工作了多少年、偏好什么语言，所以它只能给出适用于所有人的通用回答。

请记住这次回答的内容和质量——它现在"能答"，但我们还无法说它"答得好"。

```bash
agentcore invoke --runtime hrassistant \

```

!!! warning
    `session-id` 必须至少 33 个字符。上述命令会自动生成足够长度的 ID。`--stream` 启用实时输出显示。

!!! info "我都带了 --actor-id 了,它怎么还问我是谁?"
    这是预期的,别奇怪。`--actor-id` / `--user-id` **不是"告诉 Agent 你是谁",它只是 Memory(记忆)的分区钥匙**——后台用它决定"这次对话的记忆存进哪个抽屉、从哪个抽屉读",模型本身并看不到这个值,更不会自动把它解析成一份员工档案(姓名、工龄、部门)。
    
    所以当你问"我有多少天假期",会遇到两件事叠加:
    
    
    
    - **这是第一次对话,这个抽屉是空的**——Memory 里还没存过这个 actor 的任何偏好/事实,即使 actor-id 给了也没东西可读。
    
    - **查个人余额需要具体工号**——`check_leave_balance` 这类工具要一个明确的员工 ID 才能查,而 actor-id 没作为身份喂给模型,所以它只能反过来问你要。
    区分两种"知道":**政策怎么写的**来自知识库(所以年假政策那段它答得出);**你是谁、你的余额**要靠 Memory 累积或带 ID 查工具,这两样第一次对话时都还没有。等后面 Memory 攒了几轮,个性化才会显现——这恰恰是本页"裸表现基线"想让你看到的。

![](/images/placeholder-first-invoke.png)

*你实跑时看到的措辞会和这张图不一样——大模型每次回答都有随机性,这很正常。重点看两点:它给的是一份**通用**年假政策;而问到"我有多少天"时,它**反过来问你工号**——因为它还不知道你是谁,也没有查你余额所需的信息。*

### Step 2: 读这次回答,盯住两件事

它从知识库检索了年假政策,给出一段带引用的结构化回答——**读着挺专业**。但别急着满意,盯住两点:

- **答案是通用的**。它不知道你的工龄(工龄影响天数,15-20 天)、不知道你部门的高峰期、也说不出你的具体余额——因为它还没有你的任何上下文,这一步本来就该这样。

- **你没法判断它准不准**。这些政策细节,是从知识库真查来的,还是模型凭常识编的?光读回答根本看不出来。在 HR、合规这种场景,这个"看不出来"就是可信和事故的分界线。
