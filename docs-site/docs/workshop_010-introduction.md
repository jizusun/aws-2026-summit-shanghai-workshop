## 问题场景

大多数公司处理 HR 咨询的方式是这样的：

- 员工问"我有多少天年假？"——等几个小时才能得到回复

- HR 同事每周回答几十遍同样的入职清单问题

- 新员工咨询福利注册——被转接三个人才得到明确答案

- 半年后审计时被问"谁批准了这个政策例外？依据是什么？"——无法追溯

- 每次交互都从零开始——HR 对之前的对话内容和员工上下文毫无记忆

你的团队已经用大模型搭了一个能回答问题的原型。它在演示里表现不错——上面这些问题都能流畅作答。

**但"演示里表现不错"和"可信赖地服务 5000 名员工"之间，隔着一道鸿沟。** 跨过它靠的不是更强的模型，而是一套工程纪律——而这门课要讲的第一件事，是这套纪律为什么必须从 Evaluation 开始。

## Agent 与传统软件的本质差异

为什么 Agent 从原型到生产如此困难？根因在于 Agent 系统与传统软件存在三个本质差异：

| 特性 | 传统软件 | Agent 系统 |
| --- | --- | --- |
| 确定性 | 同样输入 → 同样输出 | 同样输入 → 统计上不同输出，二元 pass/fail 不适用 |
| 变更可追踪性 | 代码变更有 diff、有版本历史 | 改 Prompt 没有 diff，但行为已经变了 |
| 依赖稳定性 | 依赖版本可锁定 | 模型供应商更新会悄悄改变 Agent 行为 |

传统的 CI/CD 和 QA 框架不是为这类系统设计的。需要一套新的开发方法论。

## Agent 开发生命周期（ADLC）

这套方法论的核心是 **ADLC（Agent Development Lifecycle）**——一个持续转动的飞轮：Run → Trace → Evaluate → Diagnose → Optimize → 再循环,每一次改动都由一次重新评估来证明是否真的变好。

![](/images/adlc-flywheel.png)

关键认知：

- **ADLC 是飞轮，不是线性流水线。** 生产不是终点，而是飞轮最富价值的输入——生产中的 Trace 数据是评估数据，未来可变为训练数据。

- **"定义好"是第一步，不是上线后的复盘。** 先建立评估标准，才能客观衡量构建成果。

- **80% 的性能提升来自 Harness 层**（Prompt、工具配置、检索策略），不是换模型。Evaluation 是唯一能告诉你"改完是否真的变好"的机制。

## 企业 Agent 开发的最佳实践

Amazon 内部经验总结为 9 大最佳实践。不用一个个记——它们统一到一条主线:**要么在直接构建 Eval 体系,要么在为 Eval 奠定基础。** 在后面的动手环节里,你会自然碰到其中最关键的几条(BP1/BP2/BP3/BP4/BP8)。

如果你对完整的 9 大实践感兴趣,展开看细节:

**第一类:直接构成 Evaluation 体系**

| BP | 要点 |
| --- | --- |
| BP1: 从小做起，定义明确的成功标准 | 先问"解决什么问题"而非"Agent 能做什么"；产出 Ground Truth 数据集 |
| BP4: 从第一天开始自动化评估 | 50-100 标注案例 → 5-7 指标 → CI 门控 → 生产采样 → 持续补充 |
| BP8: 持续测试与改进 | Eval 是唯一能验证"改完有没有真的变好"的机制 |

**第二类:为 Evaluation 提供基础设施**

| BP | 要点 |
| --- | --- |
| BP2: 从第一天就埋好可观测性 | 三层：开发者层/平台层/运营层；没有 Trace，在线 Eval 无从采样 |

**第三类:让系统架构更容易被评估**

| BP | 要点 |
| --- | --- |
| BP3: 建立清晰的工具策略 | 工具定义越清晰，"工具调用正确性"评估信号越精准 |
| BP5: 用工具调用替代模型内部推理 | 外包给工具的操作可用程序性验证替代 LLM-as-Judge |
| BP6: 多 Agent 系统保持解耦 | 解耦是"Eval 可归因"的架构前提 |
| BP7: 安全从外部强制执行 | 外部策略可观测、可测试；LLM 内部推理"决策"无法被系统性验证 |
| BP9: 先把单个 Agent 做好 | 单 Agent 的 Eval 基础是多 Agent Eval 的前提 |

## Evaluation 的四重角色

Evaluation 在 ADLC 中同时承担四个角色：

- **规格说明**——定义"什么是好"，使团队对标准达成共识

- **质量门控**——判定能否上线

- **生产监控**——持续观测线上质量

- **改进驱动力**——告诉你低在哪、该改什么方向

越早构建这套能力，越能形成持续复利优势。

## 本训练营构建目标

基于以上方法论，我们将使用 AgentCore Managed Harness 构建一个企业 HR Agent，解决前述的具体问题：

| 痛点 | Agent 方案 | AgentCore 组件 | 对应 BP |
| --- | --- | --- | --- |
| 员工无法即时获得答案 | Agent 实时查询 HR 知识库 | Gateway + Lambda (Bedrock KB) | BP3 |
| 每次都要重复说明员工信息 | 自动记住部门、角色和偏好 | Memory | BP2 |
| 回答质量参差不齐 | System Prompt 约束行为 | Managed Harness | BP8 |
| 知识领域难以扩展 | 按需加载详细指令 | Skills | BP5 |
| 决策和回答不可追溯 | 每一步推理全链路记录 | Observability | BP2 |
| 无法客观衡量质量 | 自定义评估器批量评分 | AgentCore Evaluators | BP4 |

## 学习目标

完成本训练营后，你将能够：

- 使用 AgentCore CLI 创建和部署 Managed Harness

- 配置 Gateway 的 Lambda 目标以访问 Bedrock Knowledge Base

- 理解 Memory 的 user-preference strategy 如何自动提取和检索员工上下文

- 配置 Skills 实现按需的详细政策分析

- 创建自定义评估器，批量评估 Agent 质量

- 基于评估诊断定位问题根因，并用数据验证改进

- 通过 Traces 审计 Agent 的完整推理过程
