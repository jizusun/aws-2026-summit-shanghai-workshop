4.2 运行批量评估

目标

对三个代表性问题（绩效复核、福利申请、病假）的对话 Trace 跑一遍批量评估，拿到的不只是分数，而是诊断结论和改进方向。这一步会让你看到评估真正的价值：它能从一堆"读起来都不错"的回答里，精确指出哪一条是"编得好"、根因在哪、该怎么修——这些结论会直接驱动 Phase 5 的优化动作。

关于分数：你跑出来的数字会和本页示例不同

本页所有分数（GR、SP2、Pass/Fail 等）都是某次运行的示例值。THELMA / Mind the Goal 用 LLM 当裁判（本 workshop 默认 Nova 2 Lite），每次跑、不同模型，分数都会波动——你实跑时绝对值会不一样，甚至同一个问题的 Pass/Fail 都可能不同（例如福利问题有时 Pass 有时 Fail）。这是 LLM-as-judge 的正常特性，不是你操作错了。重点不在抠某个小数，而在看：维度之间的关系（如 SP1 高但 SP2 低）、诊断给出的方向、以及优化前后的变化趋势。

最省事的方式：直接运行 09-run-eval.sh——它会自动跑这三个问题生成 Trace、轮询等待 span 索引完成，再逐 trace 跑 THELMA、逐 session 跑 Mind the Goal，并打印每条的 Query、Response 与分数。

解读结果

THELMA 输出示例（一条检索质量好的对话）：

trace=6a2bf4d3dd515126 value=0.895 [Pass]
  THELMA 7 维 (query: benefits enrollment process):
  GR(接地)=0.895 | SP1(块级检索精度)=1.00 | SP2(事实级检索精度)=0.53 |
  SQC(源覆盖)=1.00 | RP(响应精度)=0.63 | RQC(响应覆盖)=1.00 | SD(去重)=0.80. 诊断: 无

对比一条检索质量差的对话：

trace=6a2bf4e8bc636ed1 value=0.389 [Fail]
  GR(接地)=0.389 | SP1(块级检索精度)=1.00 | SP2(事实级检索精度)=0.09 |
  SQC(源覆盖)=0.00 | RP(响应精度)=0.33 | RQC(响应覆盖)=1.00 | SD(去重)=0.78.
  诊断: RP↓ SP1↑ -> Prompt or Source chunking; SQC↓ RQC↑ GR↓ -> Prompt

SP1 与 SP2 —— 为什么有两个检索精度：SP1 是块级（召回的 chunk 整体相关吗？），SP2 是事实级（chunk 里的原子事实，有多少真正相关？）。上面那条差对话里 SP1=1.00 但 SP2=0.09：chunk 名义上对题，但它携带的事实里 91% 是无关噪音——正是混进政策文档的 HR-MultiWOZ FAQ 脏数据。只看 SP1 会掩盖 KB 质量问题，只有 SP2 能暴露它，这正是我们保留 SP2 的原因。

怎么读这个诊断：SQC↓（源里没覆盖问题）+ RQC↑（但回答答全了）+ GR↓（回答无源支撑）→ 结论 Agent 在用模型自身常识编答案，而非基于检索。这是企业 RAG 最危险的情况：回答看似正确但不可溯源，政策一旦变更就会答错且无法追溯。诊断直接指向修复方向：改 Prompt（强制"仅基于检索内容回答"）或改检索/源文档。

Mind the Goal 输出示例（一个"咨询"单轮会话）：

value=1.0 [Pass]
  Mind the Goal: GSR=100.0% (1/1 目标达成), 轮次数=1. 失败归因: 无失败

注意三个问题都通过了 Mind the Goal（GSR=100%）——Agent 每次确实达成了用户的目标。质量问题出在"它怎么达成的"（检索接地），这正是 THELMA 揭示的。两个评估器互补：MtG 衡量目标达成，THELMA 衡量回答可信度。

你已经跑通了第一轮批量评估，并从分数里读出了三条系统级结论和一份改进清单。现在你手里有的不再是"感觉 Agent 还行"，而是"哪一条不可信、根因在哪、下一步改什么"。Phase 5 见分晓。

不想逐条手敲上面的命令？直接运行 09-run-eval.sh 即可，它已经把这些命令都打包好了：

  • ./09-run-eval.sh —— 自动跑三个 golden 问题（绩效 / 福利 / 病假）产生 trace，再逐 trace 跑 THELMA 并 逐 session 跑 Mind the Goal，并打印每条的 Query + 截断 Response + 分数
  • ./09-run-eval.sh --eval-only [N] —— 跳过对话，直接评估最近 N 条含检索的 trace（默认 3）
  • ./09-run-eval.sh <trace-id> —— 对指定单条 trace 跑 THELMA
  • ./09-run-eval.sh <session-id> session —— 对指定 session 跑 Mind the Goal

之所以只取最近若干条而非 --days N 全量：THELMA 是 trace 级评估，--days N 会扫进大量噪音 trace（纯 HTTP/工具 span、无检索轮次），评估器只能判 Skipped，把真实分数淹没。
