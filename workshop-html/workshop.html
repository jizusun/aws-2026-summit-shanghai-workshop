从 Demo 到生产：一道工程纪律的鸿沟

做出一个能演示的 Agent，可能只需要一周。但从"能演示"到"可信赖地服务全体员工"，中间隔着一道大多数团队都低估的鸿沟——本质不是模型不够强，而是缺一套工程纪律。

原因在于 Agent 与传统软件有根本不同：行为是非确定性的，"源代码"是自然语言，依赖的基础模型会随供应商更新而悄然改变。这意味着两件事——传统测试方法失效了，而且**"好"的定义本身也变了**。

在生产环境里，一个 Agent 好不好，不再用"任务完成率"这一把尺子衡量，而是看它的决策：决策是否贴合用户与业务目标（决策质量 / Decision Quality）、能否足够快地评估并响应（响应速度 / Time-to-Action）、以及替人卸载了多少判断与重复劳动（认知卸载 / Cognitive Offload）。这是从"执行优先"到"决策优先"的转变，也是企业级 Agent 真正的价值所在。

要让一个 Agent 稳定地交付这样的决策质量，需要覆盖完整生命周期的工程纪律：界定范围、配置工具与记忆、外置安全、埋好可观测性、部署上线、持续评估与改进。而把这一切焊成一个可迭代闭环的那道焊缝，是 Evaluation：它先定义"什么是好"，再决定能否上线，再监控线上质量，再驱动下一轮改进。你无法改进你无法度量的东西——这正是贯穿全程的主线。

你将亲手走通的

本训练营基于 Amazon Bedrock AgentCore 的企业级 Agent 最佳实践，用一个 HR 问答场景，带你走完一轮完整的"构建 → 评估 → 优化"飞轮。搭建 Agent 只是载体；贯穿其中的主线——用评估把每一次改动变成可验证的进步——才是你真正带走的能力。两小时后，你带走的不只是一个能跑的 Agent，而是一套可重复、数据驱动的质量改进方法。

场景设定

你是一家企业的技术负责人，需要把 HR 咨询自动化——年假、病假、福利、绩效这些每天上百次重复的问题，原本靠 HR 同事人工回答。团队已经用大模型 + 知识库搭出了一个能回答问题的原型，初看也答得头头是道。但你心里有几个挥之不去的问题:**这些回答里的政策细节是真的查了知识库,还是模型凭"常识"编的?**这种"看起来对但说不出哪里不对"的回答,在 HR 这种合规场景里就是事故的种子。从"能回答"到"可信赖地服务全体员工",这中间隔着的距离,就是这个训练营要带你走完的路。
