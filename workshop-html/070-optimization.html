Phase 5: 优化与验证

形式: 动手实验 | 时长: 10 分钟

这一步在做什么

Phase 4 的批量评估给出了系统级诊断和改进方向。Phase 5 实际动手优化，并用同一套评估器验证改进是否真的有效。

这是 ADLC 飞轮的核心动作：不凭主观判断调参，用评估数据驱动改进。

为什么改 Prompt 而不是换模型

BP8 指出：80% 的性能提升来自 Harness 层，不是更换基础模型。改进清单里有多个方向（改 Prompt、改检索、清洗知识库），但受限于训练营时长，我们选择闭环最短的一类：优化 System Prompt。

Phase 4 的分数暴露出两类截然不同的问题：

  • 检索质量好、但回答没用好（绩效、福利）：SP1=1.0、SQC=1.0，源文档是对的，可回答里夹带了脏数据（福利那条甚至编出"你的联系方式偏好是社交媒体"）、不够聚焦。根因在 Prompt。
  • 检索本身失效（病假）：SP1=1.0 是假象、SP2≈0.09、SQC=0——召回的 chunk 名义对题，里面 90% 却是无关脏内容。根因在知识库，不在 Prompt。

这两类问题在 Phase 5 的结局会截然不同：

  • 第一类——改 Prompt 就能救回来：绩效的 SP2/RP 会明显上升，福利那条凭空捏造的"社交媒体"泄漏会彻底消失
  • 第二类——病假问题再怎么调 Prompt 也无法弥补（手里根本没有正确的源文档），分数仍然是 Fail

你可以带着这个预判去执行 5.1 和 5.2，再用数据验证它。这正是 THELMA 诊断的价值：它在 Phase 4 就精准区分了"该改 Prompt"还是"该改检索"，让优化不做无用功。

优化方法

向 System Prompt 加入抗幻觉约束：

  • 严格基于检索内容回答，不用常识编造政策细节
  • 检索不到相关内容时明确告知"暂无相关政策"，不猜测
  • 忽略检索结果中的无关内容，只引用相关部分
  • 回答简洁聚焦，避免堆砌冗余信息

步骤概览

步骤操作目的
5.1优化 System Prompt 并重新部署加入抗幻觉约束
5.2用同一问题重跑评估，对比前后分数用数据验证改进是否有效

优化的边界：本阶段只做 Prompt 优化。Phase 4 改进清单中的"改检索/源文档"属于知识库工程，超出本训练营的动手范围。但 5.2 的数据会让你明确看到：有些问题改 Prompt 确实无法解决——这本身就是 Eval-First 方法的价值所在。

下一步

进 5.1 优化 System Prompt 把抗幻觉约束写进 Prompt 重新部署,5.2 用同一套 THELMA 量化对比。一圈走完,Phase 4 做的诊断("绩效福利改 Prompt 救得回、病假救不回")会被分数变化印证或推翻——这正是 Eval-First 飞轮第一次完整闭合。
